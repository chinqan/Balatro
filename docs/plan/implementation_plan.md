# Boss-Attack Roguelike 牌組構築遊戲 — 完整規格書製作計劃

## 專案概覽

製作一款受《Balatro》啟發的 Roguelike 牌組構築遊戲的**完整遊戲製作規格書**。核心改動：將 Balatro 的「計分過門檻」機制轉化為**「對 Boss 造成攻擊傷害」**，引入 HP 系統、防禦機制與 Boss 反擊回合。

> [!IMPORTANT]
> **核心機制轉換**：`分數 = Chips × Mult` → `傷害 = 基礎攻擊力 × 傷害倍率`
> Boss 的 HP 取代了 Balatro 的「目標分數門檻」，但保留四階段結算與排版策略的精髓。

## 🛠️ 技術棧規格 (Tech Stack)

為確保跨平台兼容性與極致的視覺性能，本專案採用以下技術架構：

- **渲染引擎**：**PixiJS 8** (高效能 WebGL 2D 渲染)
- **核心平台**：**HTML5 / WebGL** (瀏覽器原生運行)
- **音訊處理**：**Web Audio API** (用於動態電音與 Pitch Shifting)
- **視覺特效**：**GLSL Shaders** (自定義卡牌材質與 CRT 後處理)
- **開發語言**：TypeScript/JavaScript (現代前端標準)

### 原型階段美術策略 (Prototype Art Strategy)

> [!TIP]
> 原型階段無需等待美術資源即可完整驗證所有遊戲機制。

- **程序化佔位圖形 (Procedural Placeholders)**：利用 `PIXI.Graphics` + `PIXI.Text` 程式碼繪製所有卡牌、Boss、UI 元素，以幾何圖形與色彩區分功能。
- **資源抽象層 (Asset Abstraction Layer)**：設計統一的 `AssetManager` 介面，原型期返回程序化圖形，正式期無縫替換為美術 Sprite，僅需修改一處設定。
- **AI 輔助素材**：利用 AI 圖像生成快速產出風格統一的臨時插圖（卡牌、Boss 立繪、圖標），加速視覺驗證。

## 設計原則（來自 NotebookLM「Balatro設計原理」）

根據 NotebookLM 的分析，本遊戲在改造時須遵循以下原則：

- **傷害映射**：`Chips → 基礎攻擊力`、`Mult → 傷害倍率`，保留「先加法後乘法」的排版黃金法則
- **Boss 防禦**：不使用傳統 RPG 的固定減傷（會破壞乘法效益），改用**機制型干擾**（如封鎖花色、限制牌型）
- **玩家防禦**：引入護盾/格擋機制，某些花色或遺物產生的 Chips 可轉化為護盾值
- **四階段結算**：完全保留，包裝為「起手式 → 武器附魔 → 被動光環 → 神器追擊」的戰鬥演出
- **懸念保留**：不顯示預期傷害，讓玩家像觀看「魯布·戈德堡機械」一樣享受傷害數字的連鎖爆炸

## 📋 分階段執行計劃

### 🛡️ 品質關卡 (Quality Gates)

每個 Phase 完成後，必須逐項檢查以下 4 道關卡。未通過的項目必須修正後才能進入下一 Phase。

#### Gate 1: 預設行為定義 (Default Behavior)
> 每個新系統/機制必須回答：「沒有任何修改時，它的行為是什麼？」

- [ ] 所有資源（護盾、Buff、Debuff）是否定義了**預設持續時間**？（每回合清零？持續到觸發？永久？）
- [ ] 遊戲開始時（Run 開始、Floor 開始、回合開始）的**初始狀態**是否明確？
- [ ] Boss 的首次行動（第一回合）是否有定義？

#### Gate 2: 邊界值與硬上限 (Boundary Values)
> 所有數值欄位必須定義「最小值 / 最大值 / 硬上限」。

- [ ] 所有可疊加的欄位（遺物欄、消耗品欄、手牌上限）是否都有**硬上限**？
- [ ] 多個來源疊加後是否可能「突破上限」？若有，是否已加入 clamp？
- [ ] 概率型效果是否定義了 `0%` 和 `100%` 時的邊界行為？

#### Gate 3: 完整枚舉 (Exhaustive Enumeration)
> 引入一組類型時，必須**列舉所有子類型**並為每一個定義效果。

- [ ] 引入的分類系統（增強、版本、封印、牌型、卡包）是否**每一種都有完整定義**？
- [ ] 數值表（面值、等級提升）是否覆蓋**所有數值範圍**（2-A、Level 1-10）？
- [ ] 商品/消耗品是否定義了**內容物與價格**？

#### Gate 4: 跨文件一致性 (Cross-File Consistency)
> 同一概念在不同文件中的**命名、數值、措辭**必須完全一致。

- [ ] 同一實體的名稱是否在所有文件中統一？（例如 Boss 名稱只能有一種寫法）
- [ ] 位置描述應使用**「結算順序中的第一個/最後一個」**而非「最左/最右」
- [ ] 不同 Phase 中引用的同一數值（遺物欄上限、出牌次數等）是否一致？
- [ ] 攻擊力/傷害等數值是否區分了不同模式（普攻 vs 技能 vs AOE）？

### Phase 1: 核心戰鬥引擎與卡牌系統 (Game Designer)

**Skill**: `balatro-game-designer/SKILL.md`
**NotebookLM 查詢重點**: 計分引擎、小丑牌機制、風險/回報

#### 步驟 1.1 — 戰鬥引擎規格書
- 定義核心傷害公式：`傷害 = 基礎攻擊力 (Base ATK) × 傷害倍率 (DMG Mult)`
- 設計四階段戰鬥結算流程（保留 Balatro 的由左至右順序）：
  - 階段一：基礎牌型 → 「起手式」（判定攻擊屬性與基礎傷害）
  - 階段二：打出的牌 → 「連擊」（逐張卡牌飛向 Boss 造成傷害）
  - 階段三：手中的牌 → 「被動光環」（鋼鐵牌/男爵等提供乘區或護盾）
  - 階段四：遺物結算 → 「神器追擊」（遺物依序加成最終傷害）
- 定義攻防回合制：玩家出牌 → Boss 反擊 → 下一回合
- 設計 Boss HP 條與護盾機制
- 設計玩家 HP、護盾與回復機制

#### 步驟 1.2 — 卡牌系統規格
- 定義六大機制類型（+ATK, +DMG, ×DMG, $$, FX, RT）並評估前/後期價值
- 新增「防禦型」機制類型（+Shield, +Heal）
- **總量定義**：遊戲預計包含 **234 張特殊卡牌/遺物**（150 神器 + 22 捲軸 + 12 靈藥 + 18 契約 + 32 永久加持）
- 為各類別設計核心機制範例。

#### 步驟 1.3 — 遺物系統規格（對應 Balatro 小丑牌）
- 設計遺物欄位與排版策略（保留由左至右結算）
- 分類遺物類型：攻擊型、防禦型、經濟型、功能型、重觸發型

#### 步驟 1.4 — 風險與報酬框架
- 定義四種風險模型（概率自毀、資源犧牲、條件鎖定、貪婪連鎖）
- 新增「HP 賭博」風險模型（犧牲生命值換取巨大傷害）

#### 步驟 1.5 — 牌庫組成追蹤與觀察系統
- 定義牌庫組成動態變化機制（花色轉換、牌庫壓縮、複製牌、同質化對機率的影響）
- 設計牌庫查看介面 (Deck Viewer)：花色分佈條、點數分佈條、卡牌縮圖排列、已用 vs 剩餘淡化
- 設計牌型等級儀表板 (Hand Type Dashboard)：顯示各牌型等級、實際 ATK/DMG Mult、已升級金色高亮
- 定義牌型等級提升曲線（Lv.1→Lv.10 完整數值表）

#### 步驟 4.3 — 戰鬥動畫 Juice 規格
- 傷害演出時間軸（由左至右連鎖 → Boss 受擊 → HP 條扣減動畫）

- 火焰/爆炸特效門檻設計

#### 步驟 4.4 — 決策流程與資訊架構
- 繪製攻防決策流程地圖（觀察 Boss 意圖 → 選牌 → 攻/防抉擇 → 排版 → 執行）
- 資訊層級金字塔（始終顯示：Boss HP + 玩家 HP + 遺物 + 手牌）

#### 步驟 4.5 — 心流與再來一局
- 設計 Boss 被擊敗的勝利演出
- 設計玩家死亡後的 ≤5 秒重啟循環
- 設計 Boss 攻擊意圖的「可預知但不精確」的懸念設計

#### 步驟 4.6 — 卡牌視覺狀態分層系統 (Card Visual State Layering)
- **Layer 1 — 增強 (Enhancement)**：定義每種增強的底紋材質
  - 玻璃牌（透明 + 裂紋）、鋼鐵牌（金屬光澤）、黃金牌（金框）、石頭牌（灰色啞光）、幸運牌（四葉草閃爍）、萬能牌（彩虹流光）
- **Layer 2 — 版本 (Edition)**：定義 GLSL Shader 的參數規格
  - 閃箔 (Foil)：微光粒子、全像 (Holo)：UV 扭曲彩虹、彩色 (Polychrome)：漸層光暈、負片 (Negative)：反色 + 噬點
- **Layer 3 — 封印 (Seal)**：定義封蟟圖標位置、顏色與動畫
  - 紅/藍/金/紫 四色封蟟，各自帶有微動畫（向下滴蟟 / 癡動）
- **Layer 4 — 觸發特效 (Trigger VFX)**：定義卡牌被計分時的動態表現
  - 正常觸發：發光 + 輕微上彈、重觸發：閃光波、碳裂：碎裂動畫 + 波、吹爬：旋轉飛入棄牌堆
- **疊加渲染規則**：增強底紋 → 版本 Shader → 封印圖標 → 觸發動畫（嚴格後畫壓前畫）

### Phase 8: 遊戲流程與畫面操作規格 (Game Flow & Screen Specs)

**Skill**: `balatro-ui-designer/SKILL.md` + `balatro-ux-architect/SKILL.md`

#### 步驟 8.1 — 畫面流程圖 (Screen Flow Diagram)
- 繪製完整的遊戲狀態轉換圖（每個畫面之間的進入/離開條件）
- 定義過場動畫類型（淡入淡出 / 滑動 / 特效轉場）

#### 步驟 8.2 — 主選單與進入遊戲
- **主選單 (Title Screen)**：按鈕佈局、背景動畫、BGM 指定
- **難度選擇 (Stake Selection)**：賭注等級解鎖條件、互動方式（點選/滑動）
- **地城地圖 (Dungeon Map)**：岥路選擇操作、房間圖示、Boss 預覽、跳過機制的互動

#### 步驟 8.3 — 戰鬥中玩家操作細節
- **手牌操作**：點擊選牌 / 拖曳排序 / 確認出牌 / 棄牌 的互動回饋
- **遺物排版**：拖動重新排序神器的操作、即時傷害預覽變化
- **Boss 意圖查看**：懸停/點擊查看 Boss 攻擊意圖的互動
- **商店畫面**：購買/售出的操作流程、重新整理 (Reroll)、離開確認

#### 步驟 8.4 — 結算與重啟畫面
- **戰鬥勝利結算**：獎勵展示順序、戰利品揀取操作、「繼續」按鈕
- **Run 失敗結算**：本局統計資料、最佳紀錄更新、解鎖進度展示
- **再來一局流程**：「快速重啟」 vs 「回主選單」的選擇、≤ 5 秒無摩擦重啟

#### 步驟 8.5 — 輔助畫面
- **暂停/設定**：音量、畫質、無障礙選項、退出確認
- **收藏庫/圖鑑**：瀏覽已發現的神器/卡牌、篩選與排序操作
- **新手教學**：分步引導、互動式提示、可跳過設計

### Phase 9: 核心基礎建設規格 (Core Infrastructure Specs)

**Skill**: `balatro-game-designer/SKILL.md`

#### 步驟 9.1 — 存檔/讀檔系統 (Save/Load System)
- **持久化方案**：定義採用 `localStorage` 或 `IndexedDB`（考慮數據量大小）
- **Run 狀態序列化**：定義哪些數據需要儲存（牌庫狀態、遺物排列、Boss 擊殺進度、商店庫存、種子值、金錢、HP、樓層）
- **自動存檔機制**：定義觸發存檔的時機（每場戰鬥結束後、進入商店時、離開遊戲時）
- **雲端存檔**（選配）：是否支援跨裝置同步

#### 步驟 9.2 — 隨機種子系統 (RNG/Seed System)
- **種子架構**：單一主種子 → 派生子種子（商店種子、卡包種子、Boss 機制種子、戰利品種子）
- **分享與復現**：玩家可複製種子碼以分享 Run、開發者可用種子碼復現 Bug
- **隨機演算法**：採用哪種 PRNG（如 Mulberry32、xoshiro128）

#### 步驟 9.3 — 牌庫管理引擎 (Deck Management Engine)
- **數據結構**：抽牌堆 (Draw Pile)、手牌 (Hand)、棄牌堆 (Discard Pile) 的狀態機
- **牌庫操作**：洗牌演算法 (Fisher-Yates)、抽牌、棄牌、刪牌（壓縮）、加牌（膨脹）
- **卡牌變化追蹤**：卡牌被增強、改變花色、附加封印等狀態變化的管理
- **牌庫組成統計**：即時計算花色分佈 (`suitDistribution`) 與點數分佈 (`rankDistribution`)，供 Deck Viewer UI 顯示
- **牌型等級追蹤**：維護 `handLevels` 狀態（各牌型等級），供 Hand Type Dashboard UI 顯示，出牌結算時從此表引用基礎值

#### 步驟 9.4 — 統計數據追蹤 (Statistics Tracking)
- **單局數據**：最高單次傷害、總傷害、最長連擊、使用最多牌型、擊殺 Boss 數、金錢收入
- **全局數據**：總 Run 次數、勝率、最高難度通關、神器發現率
- **數據流向**：統計數據如何餵入結算畫面、收藏庫、和解鎖系統

### Phase 10: 局外進度與耐玩性 (Meta-Progression & Replayability)

**Skill**: `balatro-game-designer/SKILL.md` + `balatro-level-designer/SKILL.md`

#### 步驟 10.1 — 局外解鎖系統 (Meta-Progression / Unlock System)
- **解鎖層級**：定義哪些內容需要解鎖（神器、卡包、賭注難度、起始牌組、挑戰模式）
- **解鎖條件**：擊殺特定 Boss、達成特定傷害、使用特定牌型等
- **收藏庫風格與進度條**：視覺化玩家的全遊戲探索度

#### 步驟 10.2 — 起始牌組多樣性 (Starter Decks)
- 設計 **15+ 種起始牌組**，每種從第一回合就改變遊戲策略
- 範例：「戰士牌組」（+1 出牌次數）、「亡靈牌組」（提高契約出現率）、「煉金牌組」（改變計算公式）
- 定義解鎖條件（與步驟 10.1 聯動）

#### 步驟 10.3 — 挑戰與成就系統 (Challenges & Achievements)
- 設計 **20 種挑戰模式**（特殊規則關卡）：如「禁神器」、「單牌型限定」、「不死人」
- 設計成就觸發條件與獎勵（解鎖新神器、新牌組、裝飾性項目）

#### 步驟 10.4 — 平衡性測試框架 (Balance Testing Framework)
- **數值模擬規範**：定義自動化模擬 Run 的測試腳本規格（隨機組合 N 次，統計勝率、平均傷害）
- **破局檢測**：定義「過強」的閾值（例如單次傷害超過 Boss HP 的3 倍）
- **回歸測試**：每次調整數值後，重新執行模擬以確保沒有新的破局組合

## Verification Plan

由於此計劃的交付物為**文件規格書**（非程式碼），驗證方式如下：

### 內容完整性驗證
- 每個 Phase 完成後，對照上方規格書結構表，確認所有章節均已產出
- 確認每章節的格式符合對應 Skill Agent 的模板規範

### 跨系統一致性驗證
- Game Designer 的卡牌規格文件中的數值，與 Level Designer 的 Boss HP 曲線是否匹配
- UI Designer 的色彩編碼，與 Narrative Designer 的文案風格指南中的「色彩即語法」是否一致
- Tech Artist 的 Shader 強度驅動規則，與 UI Designer 的火焰/爆炸門檻是否對應

### 人工審閱（由使用者進行）
- 每完成一個 Phase，透過 `notify_user` 提交該階段的規格書供審閱
- 使用者確認後才進入下一個 Phase
- 最終 Master GDD 需要使用者的最終確認
